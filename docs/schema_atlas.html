Schema Atlas — grid schemas against a reference model

Power-system data models · a field guide

One grid, four schemas, one ruler

Glassbox, Sienna, PyPSA, and CGMES each write down one physical power system for a different job — teaching, production analysis, optimization, exchange. Rather than rank them against each other, this atlas measures each against a reference model: what a shared grid schema could represent, assembled from the union of the field. They agree almost perfectly on the physical spine and diverge exactly where their ambitions do.

01

What a shared schema should represent

A yardstick, not a hub: no data flows through the reference model — it only enumerates what a complete grid schema could hold, so each real schema's reach reads off a fixed scale. Its capabilities are the union of the field, on the analysis ladder; the twelve principles below are each owed to the schema that argues for it best.

02

The organizing idea

Every schema needs "one system, many analyses." What differs is the seam each one cuts to get there — and that single choice explains most of everything below.

Glassbox

Tag every field

One shared Generator; each field carries the layers that read it (core / inv / ops / dyn …). A projection slices the same object per view.

one entity, faceted fields
SiennaGridDB

Split into packages

Four domains over one core — Core · Operations · Investments · Dynamics. A tool imports only the packages it needs; a thing may differ across them.

domain packages over a core
PyPSA

One flat component set

A network of typed components in DataFrames; analyses are options (optimize(), power flow), and "buildable" is a flag on the asset itself.

components + a flag
CGMES / CIM

Model the switchyard

Busbars, connectivity nodes, breakers — the physical network. The power-flow bus is computed from closed switches, not stored. An exchange ontology, analysis-agnostic.

node-breaker; the bus is derived

PyPSA stops at steady-state power flow by design — it is an optimization-first tool. Sienna adds a dynamics domain. CGMES inverts the shape entirely: it feeds none of the analyses itself, but it is the one schema native to the operations/control-room layer the others reach into last.

06

If you remember three things

On investment

The clearest fork in the road

PyPSA overloads the asset with an extendable flag; Sienna gives investment its own package; Glassbox gives it its own entities and a stepped zonal supply curve; CGMES never asks the question. Three depths and an abstention.

On modularity

Package vs. facet vs. flag

Sienna's packages are independently adoptable across a tool ecosystem; Glassbox's per-field facets make "one world, many views" literal and drive its UI; PyPSA keeps everything in one component frame and switches analysis by method.

On scope

Measure against the ideal, not each other

Each schema is an ambition: optimization interchange (PyPSA), production multi-domain modeling (Sienna), the industry exchange ontology (CGMES), a transparent teaching instrument (Glassbox). The reference model is the fixed scale that makes those ambitions comparable — and shows where each, Glassbox included, stops short.
